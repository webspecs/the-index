Abstract

This document contains the full list of documents being managed as “Web Platform Specs”, each with a minimal abstract and links to their variants.

About This Project

The manner in which Web standards are developed today is often daunting to newcomers, most notably to Web developers whose input is sorely needed in order to build a better Web.

The goal of this project is to fix this. The Web development community, and open source people in general, have over time developed work modes that are highly effective and can sustain both high quality and openness to contributions. There is no reason why such methods cannot be applied to open standards just as well, and that is what we endeavour to do.

It is still the early days and pretty much every part of the system has rough edges — but every part is also being improved. If you want to join the fun, we welcome contributions on all aspects.

You can read more in the (ongoing) documentation.

    How this works

    Specifications get added to this document through pull requests. If you wish to publish a fork of an existing specification, simply add an entry to the <script type="application/webspec+json"> block you will find in the source. The key to the object is structured as username/reponame#branch, and its value is an object with at least a description key. Keep it short and simple; as well as respectful of the other forks.

    In addition to the description key, if your file name is not index.bs (for Bikeshed) or index.src.html (for ReSpec) you can set the fileName field to the base name you are using (without the extension). Note however that this is strongly discouraged, and that the publication will always output to index.html. The feature is essentially available to repositories that have a pre-WebSpecs history and need to remain compatible with a previous setup for some reason.

    If you are creating a new WebSpec from scratch (as opposed to forking one) then you will want to make a completely new section. Simply copy from an existing one and try to place it somewhere logical in the document.

    Each specification listed here follows the same template outlined below. 
    <section>
  <h3 id="example">The Example Spec</h3>
  <p>
    This specification is a permanent example of how a WebSpec is made.
  </p>
  <script type="application/webspec+json">
  {
    "darobin/example-spec#with-toaster": {
        "description":  "Examples are always better with toasters. This part here explains what is special about the proposal."
    }
  , "sideshowbarker/example-spec#more-fun": {
        "description":  "This wasn't fun enough so I spruced it up."
    }
  }
  </script>
</section>

    When your PR is reviewed, in order to ensure that your fork is automatically published whenever you make changes to it you just need to create a Web hook in your GitHub repository that points to http://publican.berjon.com/hook. Once your PR is accepted, the WebSpecs index will be updated and your specification will be published (and republished automatically if you’ve got the hook setup).
